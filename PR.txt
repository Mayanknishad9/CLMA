# PR: AI 闭环自进化工作流全栈系统（可热更新插件架构 · 社区生态预备）

## 概述
构建一套**通用、自进化、高度可扩展的闭环推理验证系统**，融合 C++ 高性能引擎与 Python 美观交互界面。系统以经典 **Planner → Executor → Verifier → Critic** 闭环为核心，采用**全插件架构**，支持工具、规划策略、修正器、评判标准等模块的**热插拔与动态升级**。用户可通过 UI 一键安装、卸载社区开发的插件，无需重启引擎即可生效。同时提供开环/闭环双模式、多工具接入、评判规则自定义、全流程可视化监控、会话级数据管理和 LLM 模型列表动态适配。

---

## 核心设计理念 —— 插件化 · 热更新 · 社区友好
所有核心功能均构建在 **动态加载 + 接口抽象** 的插件体系之上，系统行为可随时扩展而无需修改引擎源码。

- **工具插件化**：任何满足 `Tool` 接口的动态库（.so/.dll）或 Python 模块均可被引擎发现、加载和卸载。内置工具同样遵循该接口。
- **策略插件化**：Planner 的推理策略、Critic 的修正策略、Verifier 的评判方法均定义为可替换插件，用户可编写自己的策略并上传分享。
- **热更新机制**：C++ 引擎内置插件目录监控，检测到新增或更新文件后自动重载，无需停止服务。Python 交互层提供插件市场 UI，一键获取社区资源。
- **社区开发支持**：提供清晰的插件开发模板、文档和测试套件。用户上传的插件经安全审查后可在市场发布，其他用户可直接安装使用。
- **安全隔离**：每个插件在独立沙箱中运行，权限细粒度控制，防止恶意代码影响主机。

## 功能详设

### 1. 优化后的 Agent 分工与协作（插件化实现）
所有智能体角色均通过插件组合实现，核心引擎仅负责调度和状态管理。

| 角色 | 插件化实现 | 热替换场景 |
|------|------------|------------|
| **Planner** | 加载 `IPlannerStrategy` 插件，内置默认策略，可选社区提供的“多步推理”、“DAG规划”等策略 | 任务类型变化时自动切换规划算法 |
| **Executor** | 调用 LLM，支持并行生成 K 个候选。候选混合策略（温度调整、投票）以插件形式提供 | 根据不同模型或实验需求更换采样器 |
| **Verifier** | 硬验证加载工具插件（编译器、计算器等）；软验证加载 `IJudge` 插件，执行用户定义的评分标准 | 随时添加新的验证工具或升级评估模型 |
| **Critic** | 加载 `ICorrector` 插件链（如正则修正、AST 补丁、LLM 重写），按序执行 | 针对不同语言或错误类型灵活组合修正器 |
| **Meta-Verifier** | 可选插件，评估计划合理性 | 社区开发更严格的校验规则 |
| **Experience Cache** | 缓存后端插件（本地 JSON、向量数据库等） | 无缝切换存储方案 |

### 2. 闭环优化方案（插件驱动）
- **并行竞赛**：候选生成器和验证并行度可通过插件参数调节。
- **分层修正**：结构化解构与局部定位功能由 Critic 插件实现，支持多语言/多格式扩展。
- **智能终止**：终止条件插件可自定义，如连续失败次数、资源阈值等。
- **经验复用**：缓存匹配算法插件化，默认提供向量相似度，可替换为其他检索策略。

### 3. C++ 核心引擎（插件运行时）
- **插件管理器（PluginManager）**：负责动态库扫描、加载、符号解析、生命周期管理、依赖检测和热重载，支持版本兼容检查。
- **沙箱环境**：每个插件在独立进程或 Docker 容器中执行，权限受限，文件系统和网络调用受白名单控制。
- **状态机**：基于插件节点构建，新增验证步骤只需实现 `ILoopState` 接口并注册到配置。
- **Python 绑定**：导出 `core_engine.plugin_manager` 供 Python 层调用，支持安装、卸载、启用/禁用插件。

### 4. Python 交互层（插件市场与自定义开发）
- **插件管理界面**：
  - 内置“插件市场”标签页，展示官方和社区验证的插件列表，支持一键安装/卸载。
  - 本地开发插件可直接上传并加载，开发过程中提供实时测试面板。
  - 插件更新通知和版本回滚功能。
- **动态模型适配**：通过 `ModelProvider` 插件对接不同厂商 API，新增厂商只需安装相应插件，同时自动拉取可用模型列表。
- **存储与可视化**：会话存储继承插件数据格式，支持插件自定义数据持久化，加入临时存储区，记录每一次迭代输出的信息，直接本地写入，并且校验后输出，以及最终结果存储，同样是本地直接写入，校验完成后输出，保存用户对话历史，新一轮开始后删除临时存储记录，只保留最终结果信息。；流程图动态反映插件加载的额外步骤，也就是在推理运行时用可视化流程图表示每一个agent的实时交互情况，而不是让用户静静等待，看着转圈圈，当发生错误也可以直接在流程图上实时反馈信息。

插入对话补充，框架进一步实现：
这个框架的**核心设计（规划-执行-验证-修正循环 + 可注册工具 + 自主推演验证需求）已经抓住了自迭代智能体系统的本质**，模块解耦清晰，语言分工也合理。但若要真正落地为“高效、可靠、可扩展”的工程产品，以下方向值得进一步优化。

---

### 一、框架优点（已做对的事）
- **闭环控制逻辑独立于业务**：通过工具注册和 Planner 的推理实现通用性。
- **角色分离**：Planner、Executor、Verifier、Critic 各司其职，便于独立升级。
- **支持不同验证模式**：规则工具（编译器/计算器）与 LLM-as-judge 可以共存。

---

### 二、需要进一步打磨的关键点

#### 1. Planner 的推理质量与稳健性
**挑战**：AI 生成的验证计划可能不完整（遗漏关键验证）、过于激进（验证所有东西导致效率低）或产生幻觉工具名。  
**改进建议**：
- **增加“计划验证-计划”的元认知步骤**：让 Planner 自检其验证计划是否覆盖了所有可能的失败模式（通过 few-shot 示例强化）。
- **工具模式强制约束**：使用 structured JSON 输出 + Pydantic 校验，过滤无效工具调用。
- **引入类似 Chain-of-Verification 的多步推理**：先生成初步计划，然后让同一个模型逐项确认必要性，再精简。

#### 2. 收敛效率与成本控制
**挑战**：循环多次调用 LLM（Planner、Critic、可能还有 Verifier）既费钱又慢，且可能因 Critic 给出错误修正而发散。  
**改进建议**：
- **并行化探索**：在同一轮迭代中让 Executor 生成多个候选输出，Verifier 并行评估，择优进入下一轮或直接结束，减少串行轮次。
- **Critic 的反馈压缩**：要求 Critic 直接输出“具体的修改指令”而非自然语言评价，Executor 可按指令定位修改，提升一次修正成功率。
- **增加提前终止条件**：若连续两轮验证结果无变化或冲突度上升，强制中止并请求人类干预，避免死循环。
- **缓存与记忆**：将成功解决过的问题的 `(请求, 计划, 验证通过输出)` 存入向量库，相似问题直接复用计划，跳过推理。

#### 3. 验证的粒度与可靠性断层
**挑战**：不是所有任务都有二元的通过/失败判定，比如“写一篇软文好不好”缺乏明确验证工具。  
**改进建议**：
- **混合验证堆栈**：
  - *硬验证*：代码编译、数学计算、格式校验（必须 100% 通过）。
  - *软验证*：LLM 依据评分标准打分（如 1-10 分），设置可接受阈值（如 >7 分）。
- **评判智能体的校准**：若使用 LLM-as-judge，定期用人工标注数据校准其评分与人类判断的一致性，避免评判漂移。
- **多维度评判**：让 Verifier 输出“功能正确性”“性能指标”“安全风险”等多个维度的通过/失败，Critic 可更有针对性地修正。

#### 4. 更复杂的任务依赖与状态管理
**挑战**：当前流程是线性循环（生成 -> 整体验证 -> 整体修正），但真实问题可能需要分步验证、局部回退（比如一段代码只有一个函数错误，不需要重新生成全部代码）。  
**改进建议**：
- **版本化草稿与分层验证**：让 Executor 输出结构化文档（如带有函数分块的代码），Verifier 能定位到具体出错块，Critic 只对该块提出修正，其余不变。
- **有向无环图（DAG）规划**：允许 Planner 生成包含多个子任务和依赖关系的计划，其中每个子任务可独立验证，减少大面积重试。

#### 5. 安全与资源管控
**挑战**：开放工具调用（尤其是代码执行、文件写入）存在巨大风险。  
**改进建议**：
- **沙箱分级**：给每个工具设定风险等级，高风险工具（如 run_code）必须在隔离容器中执行，并限制 CPU/内存/网络/时间。
- **权限审批机制**：对敏感操作（写文件到关键目录）要求人类回环确认，或在 C++ 调度层做路径白名单拦截。
- **工具调用失败的状态回滚**：若某个工具调用损坏了环境，需要有能力回滚到上一稳定状态。

#### 6. 自我进化与长期学习
**挑战**：框架目前没有从历史运行中学习的能力，每次遇到相似问题都从零推理。  
**改进建议**：
- **经验库**：记录每次闭环的轨迹（规划、验证失败点、最终成功方案），抽象成 case-based reasoning 的知识条目。
- **自动工具创造**：当 Planner 表示“缺少某类验证工具”时，记录该需求，人工或自动化补充工具并注册，不断扩大系统能力边界。

#### 7. 与现有框架的集成对比
相较于 CrewAI 等固定 DAG 流程，本框架的**动态验证闭环**是更先进的形态。但 CrewAI 的 Agent 复用、任务委托等机制值得借鉴，可实现**混合模式**：对于简单问题走直通模式（开环），复杂问题自动升入闭环，并由同一个 Manager Agent 调度，体验更统一。

---

### 三、优先实施路线
如果要挑选最能提升系统鲁棒性和实用性的改进：
1. **强制工具参数验证 + Planner 自检**（防幻觉，易实现）
2. **并行生成多个候选 + 择优修正**（提效显著）
3. **硬验证/软验证组合 + 可分块修正**（拓宽适用场景）
4. **沙箱与权限分层**（安全必须前置）

这些优化会让框架从一个精巧的“概念原型”真正走向可生产部署的“AI 工场”。
